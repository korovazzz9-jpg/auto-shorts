# 60SecFacts / Datos en 30s — автоматический YouTube Shorts

Два параллельных канала на одном коде:
- **60SecFacts** (EN) — английский, 4 Shorts/день + 1 лонгформ/неделю, Instagram включён
- **Datos en 30s** (ES) — испанский, 4 Shorts/день, Instagram пока отключён

Конвейер: тема → сценарий (Claude `claude-sonnet-4-6`) → озвучка (Shorts/серии: edge-tts бесплатно; лонгформ: novita.ai MiniMax speech-02-turbo, платно, с edge-tts фолбэком) → видео (стоковые клипы Pexels/Pixabay + караоке-субтитры, MoviePy) → YouTube → Instagram (только EN).

Плюс третий пилотный поток: **вьетнамский TikTok** (`Sự Thật 60 Giây`) — random-facts поверх залипательного фона, публикуется вручную (см. раздел «TikTok» → «Вьетнамский»).

## Как работает пайплайн (важные детали)

### Выбор темы (`generate_script.py`)
14 тем-пулов (space, ocean, history и т.д.). Пока данных меньше чем по 5 темам — выбирает случайно. Скрытый тег `topic-<тема>` добавляется к каждому видео для трекинга.

**Квоты 70/20/10 (2026-07-02, вместо чистого взвешивания):** раньше темы взвешивались по средним просмотрам (`topic_stats.py`) — победители самоусиливались до усталости темы (локальный максимум: серия почти одинаковых «shipwreck»-роликов). Теперь темы ранжируются по avg views и делятся на трети: **70%** выборов — верхняя треть (проверенные), **20%** — средняя (середняки/новые, у тем без данных средний вес), **10%** — нижняя (wildcard-исследование). Внутри яруса — равномерный выбор. Лог пишет `tier=top/mid/wild` для отладки. Откат: вернуть `random.choices(pool, weights=...)` в `_pick_topic()`.

**Стоп-темы (запрещено):** физика (`physics`) и квантовая физика (`quantum physics`) — слишком абстрактны, аудитория не чувствует сюрприза без специальных знаний. Запрет прописан в `BANNED_TOPICS` в коде и в системном промпте Claude.

### Дедупликация тем (`recent_titles.py`)
Перед генерацией сценария Claude получает список последних 100 заголовков канала и не должен повторять конкретные факты.

**Дедуп ТЕМЫ (не только факта):** темы последних 2 видео исключаются из выбора следующего (`get_recent_topics` → `_pick_topic` фильтрует пул), чтобы не выходило два похожих ролика подряд (напр. два «shipwreck»). Тема пишется в `topics_cache.json` сразу при генерации (обходит лаг индексации YouTube), кеш персистится через `actions/cache`. Серии тоже пишут свою тему в кеш — daily её не повторит следующим роликом.

⚠️ **Баг, живший до 2026-07-02:** ключ кеша был статичным (`titles-cache-en`), а `actions/cache` при точном совпадении ключа НЕ пересохраняет кеш в конце джобы — т.е. кеш записался один раз при первом запуске и с тех пор не пополнялся: защита «не 2 темы подряд» в CI фактически не работала, а `prepare_batch.py` (без YT-креды) строил avoid-list по замороженному кешу — вероятная причина серии почти одинаковых «shipwreck»-фактов. Фикс: ключ уникален на каждый запуск (`titles-cache-en-${{ github.run_id }}`) + `restore-keys: titles-cache-en` как префикс — каждый ран восстанавливает самый свежий кеш и сохраняет обновлённый. Откат: вернуть статичный ключ (но тогда кеш снова замёрзнет).

**Жёсткий дедуп batch-очереди против уже ОПУБЛИКОВАННЫХ видео (2026-07-04, `prepare_batch.py`):** дублей ловил только `_is_duplicate()` — но он сравнивает новый элемент батча с тем, что ещё лежит в `queue_<channel>.json`, а как только видео опубликовано и уходит из очереди, сравнивать уже не с чем. Единственной защитой от повтора уже вышедшего факта оставался мягкий `avoid_block` (100 заголовков текстом в промпт) — модель не обязана его соблюдать (реальный случай: серия про бессмертную медузу вышла, через несколько дней та же тема снова прошла генерацию под другим заголовком). Фикс: `recent_titles.get_recent_video_texts(n=50)` тем же `playlistItems.list` достаёт `description` последних N видео (там лежит сам `script`, см. `publish.py`), каждый новый item батча сверяется по той же сигнатуре имён/чисел (`_signature_from_text`, ≥2 совпадения = дубль) не только с очередью, но и с этими N опубликованными. Не требует новых вызовов квоты (те же данные, что и `get_recent_titles`). ⚠️ Для работы проверке нужны YT-креды в `prepare-batch.yml` (добавлены 2026-07-04, изначально забыты — без них фетч молча возвращал пусто и проверка была no-op; теперь prepare_batch печатает «Дедуп против опубликованных: N видео загружено», N=0 — сигнал проблемы с кредами). Откат: убрать проверку `published_signatures` в `prepare_batch.py`, оставить только `_is_duplicate` против очереди.

### Голоса TTS (`tts.py`)
Ротируются между видео (EN: 5 голосов, ES: 4 голоса) — YouTube следит за "inauthentic content", одинаковый голос каждый раз — красный флаг.

### Стоковое видео (`fetch_stock_video.py`)
Сначала Pexels, fallback на Pixabay. Claude генерирует поисковые запросы под каждый визуальный beat сценария (1 бит ≈ каждые 4-5 секунд), запросы пишутся под **настроение/обстановку**, а не под конкретный объект (стоки не имеют нишевых артефактов).

Три механизма качества клипов:
1. **Vision-отбор** — на каждый запрос берётся до 4 кандидатов, в Claude Haiku отправляются их **poster-кадры по URL** (Pexels отдаёт `image` в ответе — видео НЕ скачиваются), Haiku выбирает релевантный, полностью качается только победитель. ~$0.003/видео, трафик минимальный. Если у кандидатов нет preview (Pixabay-fallback) — берётся первый по релевантности стока. `VISION_CANDIDATES = 4`.

**Vision-кэш (2026-07-02, `vision_cache.json`):** стоковые запросы повторяются между видео («ocean waves», «ancient ruins»…) — выбор Haiku кэшируется на 7 дней (`запрос → id одобренного клипа`), повторный запрос не дёргает vision вообще. Vision — главный Claude-расход после генерации скриптов, кэш срезает повторные вызовы с нулевым риском (тот же запрос → тот же уже одобренный клип). Хит валиден, только если клип есть среди текущих кандидатов (used_ids-фильтр внутри видео уважается). Файл персистится через `actions/cache` вместе с titles/topics. Откат: удалить `vision_cache.json` из path кэша в воркфлоу — файл перестанет переживать раны, каждый запрос снова через Haiku.
2. **Дедупликация** — `used_ids` в рамках одного видео: один и тот же сток-клип не попадёт дважды.
3. **Упрощение запроса** — если по полному запросу клипов нет, пробуем первые 2 слова (`mantis shrimp moving underwater` → `mantis shrimp`).

Фильтр: только вертикальные (`height >= width`), высота ≥ 960px. Pexels API запрашивается с `orientation=portrait`.

**Satisfying-фон (VN TikTok):** `SATISFYING_QUERIES` + `fetch_satisfying_clips()` — отдельный пул запросов (kinetic sand, soap cutting, fluid art и т.п.), фон НЕ тематический, все проверены на Pexels (коммерческая лицензия, без атрибуции). Используется только с `CFG["satisfying_mode"]` (сейчас — `vi`).

### Хук: шаблоны, двойной хук, плашка, тумба, стиль
Ключевой рычаг retention — первые 2-3 секунды. Всё, что относится к хуку, собрано здесь (используется и в `generate_script.py`, и в `build_video.py`):

- **5 шаблонов хука** (`HOOK_TEMPLATES` в `generate_script.py`: vague-ability / you-experience / one-category / authority-cant-explain / sounds-fake / other) — модель сообщает, каким пользовалась; пайплайн тегирует видео `hook-<template>`, `analytics_retention.py` меряет retention по типу.
- **Петля хук-аналитики замкнута (2026-07-02):** `weekly_report.py` раз в неделю пишет лучший по retention шаблон (минимум 5 видео на шаблон, «other» исключён) в `hook_stats_<channel>.json`, воркфлоу `weekly-report.yml` коммитит его в репо (тот же паттерн, что queue). `generate_script._hook_preference()` читает файл и добавляет в промпт **мягкую** подсказку «prefer template X when it fits naturally» — подсказка, не приказ, иначе шаблон станет формулой и не с чем будет сравнивать. Нет файла/мало данных — промпт как раньше. Откат: удалить `hook_stats_*.json` из репо (или сам вызов `_hook_preference()`).
- **Двойной хук** — отдельно от произносимой первой фразы модель отдаёт `hook_text` (3-6 слов, другой ракурс, НЕ копия заголовка). Глаз и ухо получают две разные зацепки в первые 2 секунды. Фолбэк на `title`, если поле пустое.
- **Хук-плашка** (`build_video._hook_clips`) — крупный статичный текст (`hook_text` или заголовок) на **тёмной полупрозрачной плашке от края до края** (без скругления и обводки букв, `HOOK_BOX_OPACITY = 0.4`), поверх первых ~2.8 сек — стоп-скролл для тех, кто смотрит без звука. Шрифт — `SUBTITLE_FONT`, по умолчанию **Anton** (`assets/fonts/Anton.ttf`, лежит в репо → работает на CI/Ubuntu), переопределяется env-переменной.
- **Тумба Shorts/серий** — кадр на `t=0.05с` (не середина хука — иначе на тумбу попадало первое слово субтитров).
- **Тумба лонгформа** — отдельно, короткая фраза `thumb_text` (3-5 слов, фолбэк на первые 4 слова заголовка) рисуется крупно на затемнённой нижней полосе — не весь длинный заголовок.
- **Стиль против «ИИ-шного» звучания** — промпт требует сильные глаголы вместо «is»/«there is» («This cave BREATHES» не «There is a cave that breathes»), запрещает generic-хеджинг («fascinating», «scientists discovered», «this phenomenon»).

### Сборка видео (`build_video.py`)
- Формат: 1080×1920 (вертикаль)
- Zoom-эффект на фоне (1.05–1.15×), рандомится
- Караоке-субтитры: слово-по-слову, 3 цвета рандомно (белый/жёлтый/мятный), идут до самого конца (не обрываются перед loop-фразой)
- CTA-пульс (сердечко + текст) в последние 2 сек, позиция ~24% сверху
- **Фоновая музыка** (`_music_layer`/`_build_audio`): случайный луп из `assets/music/*.mp3`, нормализован, зациклен под длину, `MUSIC_LOOP_VOLUME = 0.12`
- **Обрезка хвоста тишины:** перед зацикливанием режем тишину TTS после последнего слова (`duration = words[-1]["end"] + LOOP_TAIL_PAD`, `LOOP_TAIL_PAD = 0.18`) — иначе перед повтором видна пауза (~0.8с)
- **Последний клип** зацикливается если короче своего слота; остальные клипы НЕ зацикливаются — чтобы не было повторов в середине
- **Визуальный loop-хвост** (первый клип дублируется в конец, для бесшовной петли пересмотра) — только для daily (`visual_loop=True` по умолчанию); для серий **отключён** (`visual_loop=(part is None)`), см. раздел про серии
- Хук-плашка и тумба — см. раздел «Хук» выше

### Плейлисты (`playlists.py`)
Каждое обычное видео автоматически добавляется в тематический плейлист по скрытому тегу (`get_or_create_playlist` / `add_video_to_playlist`). Для серий — отдельные функции `create_playlist`/`add_video_to_playlist_by_id` (один плейлист на цикл серии, см. раздел про серии).

### Недельная серия (`pipeline_series.py` + `generate_series.py`)

**На обоих каналах.** 3 тематически связанных видео (Пн/Вт/Ср в прайм-слот 00:07 UTC EN / 00:17 UTC ES), **замещают** этот слот обычного daily, а не добавляются сверху. Один и тот же `pipeline_series.py`, канал выбирается `CHANNEL` env:
- **Part 1:** генерирует все 3 скрипта за один вызов Claude → сохраняет в `series_state_<channel>.json` → публикует Part 1. CTA: *"Follow so you don't miss Part 2"*
- **Part 2:** читает state, публикует. CTA: *"Follow for Part 3"*
- **Part 3:** развязка + *"Follow for more series like this"*

**Часть определяется по ПРОГРЕССУ серии, не по дню недели** (`_determine_part()` читает `next_part` из state; ручной override — `SERIES_PART=1/2/3`). Раньше номер части брался из `date +%u` — ломалось, если cron срабатывал не в тот день/час (было: два дня подряд оба посчитались Part 1 → перегенерация → краш на JSON). State коммитится **каждый** запуск, не только Part 1.

**Хук-фичи серий выровнены с daily:** каждая часть получает `hook_text` и тег `hook-<template>` (см. раздел «Хук» выше). Раньше `build_video()` для серий вызывался без `hook_text` — на плашке шёл весь длинный заголовок.

**Визуальный loop-хвост ОТКЛЮЧЁН для серий**: у daily в конце молчаливый follow-бейдж (+ иногда петля), у серии — **озвученный** CTA («follow for Part 2/3»); зацикливать видео на начало после него бессмысленно и режет слух.

**Навигация между частями:** Part 1 создаёт **плейлист серии** (id в `state.playlist_id`), каждая часть добавляется в него. Закреп-коммент каждой части → ссылка на плейлист; **Part 2 и Part 3** дополнительно дают прямые ссылки на все уже вышедшие части (`state.partN_video_id`, было только у Part 3 до 2026-07-01 — расширено на Part 2, чтобы зритель нашёл Part 1 сразу, а не ждал плейлиста). Решает проблему «зритель увидел Part 2/3 — как найти предыдущие в бесконечной ленте Shorts».

**Устойчивость к битому JSON:** генерация 3 скриптов — самый большой JSON-пейлоад пайплайна; иногда Claude возвращает невалидный JSON. `generate_series` ретраит парсинг до 3 раз перед тем, как упасть.

Серии идут БЕЗ loop-петли (loop только в daily Shorts, см. раздел «Loop»). Каждое видео начинается с оверлея **"PART N / 3"** (2.5 сек).

**Отключить серии** — задизейблить `weekly-series.yml` / `weekly-series-es.yml`. Ежедневные видео работают независимо.

**Запустить вручную:** Actions → Weekly Series (EN/ES) → Run workflow → part (1/2/3, пусто = авто по прогрессу). Для Part 2/3 нужен `series_state_<channel>.json` с предыдущей частью.

### Лонгформ (`pipeline_longform.py` + `generate_longform_script.py`)
Еженедельная компиляция, 3.5-5 мин. Второй путь к монетизации (1000 подписчиков + 4000 часов обычных просмотров, независимо от порога Shorts). **На обоих каналах** (генератор параметризован под `CFG`). Выходит в вечерний прайм Вс (23:00 EN / 23:17 ES UTC), отдельным слотом — шортсы не трогает.

**3 чередующихся формата** (`FORMAT_CYCLE` в `generate_longform_script.py`) вместо старого «5 фактов»-листикла (тот проваливался: ~4 просмотра, нет единой интриги):
- `deep_dive` — один факт, полный нарратив (хронология, персонажи)
- `mystery` — открывает с результата, разматывает причину до развязки
- `versus` — два явления/эпохи в контрасте до конца

Ротация хранится в `longform_state_<channel>.json` (`next_format_index`), одно значение на канал — цикл не привязан к дате, просто следующий формат при каждом запуске.

**Ретрай битого JSON (2026-07-02):** генерация лонгформа ретраит парсинг до 3 раз (как в сериях, где это добавлено после реального прод-падения) + проверяет обязательные поля (`title`/`script`/`video_queries`) внутри той же попытки — провал парсинга у лонгформа стоил бы слота на неделю. Также `thumb_text: null` от модели больше не роняет генерацию (`str(... or "")`). В сериях аналогично: проверка структуры (`part1..3` с `title`+`script`) перенесена ВНУТРЬ retry-цикла (раньше валидный JSON без `part2` падал `KeyError` уже после ретраев), `tags` не-списком коэрсится в пустой список.

Добавлен тот же ретрай на 3 попытки вокруг первичного парсинга (до отдельного validation-retry, который остался как был). Откат: убрать цикл `for attempt in range(3)` в `generate_script()`, вернуть одиночный вызов `_parse_response(message)`.

**Озвучка через novita.ai (MiniMax speech-02-turbo)** — качество голоса важнее, чем у 30-сек Shorts, edge-tts на 4+ минутах звучит механически. `tts_novita.py`: скрипт режется на **предложения**, каждое озвучивается отдельным API-вызовом, границы между предложениями = **реальная измеренная длительность аудио** сегмента (не оценка) — так исправлен рассинхрон, который давала прежняя версия (одна пропорциональная оценка на весь 4-мин скрипт, ошибка копилась к концу видео). Сегменты склеиваются в один mp3. Управляется `CFG["longform_use_novita"]` + `NOVITA_KEY`; **фолбэк на edge-tts** при любой ошибке novita — лонгформ никогда не падает из-за TTS. Голос — `CFG["novita_voice"]` (сейчас `Deep_Voice_Man`), язык — `CFG["novita_language"]`.

**Тумба и закреп-коммент** — см. раздел «Хук» (короткая фраза `thumb_text`) и `CFG["longform_comment"]` (провокационный вопрос + подписка + само-ответ; раньше `pipeline_longform.py` вообще не постил коммент).

**Горизонтальный формат (16:9, 1920×1080)** — лонгформ собирается ОТДЕЛЬНЫМ билдером `build_longform_video.py`, а НЕ вертикальным `build_video` (Shorts). Длинные видео смотрят с десктопа/ТВ, где вертикаль идёт с чёрными полосами. Билдер упрощённый: фон из клипов + лёгкий зум + караоке-субтитры снизу, без Shorts-фишек (хук-плашка, CTA-пульс, петля, PART), визуального loop-хвоста тоже нет — своя `_build_background` без дублирования клипа. Клипы качаются горизонтальные: `fetch_clips(..., landscape=True)` (Pexels `orientation=landscape`, фильтр `width >= height`).

### Главы лонгформа / key moments (2026-07-03, `_chapter_lines` в `pipeline_longform.py`)
Генератор лонгформа дополнительно отдаёт `chapters` (4-6 глав: название + `sentence_index` — индекс предложения скрипта, с которого глава начинается). После озвучки индексы маппятся на **реальные TTS-тайминги** (кумулятивный счётчик слов по предложениям → `words[i]["start"]`, дрейф ±2-3с некритичен) и вставляются в описание блоком `Chapters:`/`Capítulos:` в формате `M:SS Название`. YouTube показывает главы как key moments в поиске — бесплатный SEO/навигация для лонгформа. Требования YouTube соблюдены кодом: первая метка всегда 0:00, главы ближе 10с отбрасываются, меньше 3 валидных глав → блок не вставляется вообще (описание как раньше); любые кривые данные от модели (не список, индекс за пределами, мусор) не роняют публикацию. Откат: убрать вызов `_chapter_lines` из `pipeline_longform.py`.

### CTA (call-to-action)

Последние 2 сек каждого видео: пульсирующее сердце + pill-бэйдж с текстом.

- **Сердце** — кубические безье (`_draw_heart_png`), рендерится в 600px → downscale до 220px при анимации (нет пикселизации). Цвет `#FF1744`, тонкая тёмная тень.
- **Бэйдж** — PIL pill с тёмной полупрозрачной подложкой (rgba 0,0,0,175), белый текст. Читается на любом фоне.
- **Текст бейджа** — topic-aware: если для темы есть слово в `topic_cta_words` → персональный призыв («FOLLOW for more OCEAN facts»), иначе генерик из `cta_phrases`. Конвертит лучше генерика. Правится в `config.py` (`cta_topic_template` + `topic_cta_words` + `cta_phrases`).
- **Бейдж говорит «Follow»** (кросспостится в Instagram Reels, там кнопка Follow), но **закреп-коммент говорит «Subscribe»** (это YouTube-специфичный текст, не уходит в IG — должен совпадать с кнопкой под видео).

### Структура скрипта (`generate_script.py`, `BASE_SYSTEM_PROMPT`)
Жёсткий порядок предложений (используется daily + series через общий `BASE_SYSTEM_PROMPT`):
1. **Хук** — интрига БЕЗ раскрытия субъекта. Субъект называется только во 2-м предложении. Подробности про хук-шаблоны и двойной хук — см. раздел «Хук» выше.
2. **Reveal + факт** — обязан содержать **конкретный якорь**: число/дату/место/имя (*"100,000 years"*, *"1888 Ritter Island"*). Где честно можно — привязка к телу/жизни/безопасности зрителя.
3. **Ре-хук + твист** — короткий ре-хук (3-5 слов, «But here's the strange part») перед пейоффом, чтобы зритель не отваливался в середине, затем твист — момент, где заблуждение явно рушится.
4. **Comment bait** — один из **5** механизмов (не generic "what do you think?"): (a) ловушка-коррекция; (b) призыв к личному опыту; (c) деление на лагеря; (d) незавершённое "actually"; (e) **share-триггер** — «Send this to someone who…» (репост/share-to-DM — топ-сигнал раздачи и в YouTube, и в TikTok). **Это ПОСЛЕДНЯЯ озвученная фраза.**
5. **Голосового CTA НЕТ (daily).** Comment bait сразу перетекает в loop-фразу. Призыв подписаться — только визуально (сердечко + бейдж). Loop-строку Claude НЕ пишет — её детерминированно дописывает код. Серии — исключение: у них есть озвученный CTA («follow for Part 2»), т.к. петли нет.

**Само-ответ в комменте:** `publish()` отвечает на свой же закреп-коммент (`CFG["first_comment_reply"]`) — мини-тред, engagement density как сигнал ранжирования. То же в лонгформе.

**Пул для строки подписки в закреп-комменте (2026-07-05, `first_comment_subscribe_ctas`):** обнаружено при разборе внезапного падения просмотров EN — строка «Subscribe for a new fact every day 👉 ...» была ДОСЛОВНО одинаковой на 100% видео канала с самого начала (в отличие от само-ответа, у которого уже был пул). Идентичный текст под каждым видео — классический сигнал "bot-like pattern" для спам-детекции YouTube. `publish()` теперь заменяет вторую строку шаблона случайной фразой из пула (тот же паттерн, что `first_comment_replies`); пустой пул в конфиге → строка как раньше. Первая строка (вопрос) уже заменяется фактоспецифичным `comment_question` (см. выше). Откат: убрать блок `subscribe_ctas` в `publish.py`.

**Фактоспецифичный закреп-коммент (2026-07-04):** модель отдаёт `comment_question` в том же вызове генерации (daily + серии) — провокационный вопрос про КОНКРЕТНЫЙ факт видео (не generic «ты это знал?»). `publish()` заменяет им первую строку шаблона `first_comment`, строка подписки остаётся. Пустое/отсутствующее поле → шаблон как раньше (старые элементы очереди без поля работают без изменений). Откат: убрать замену `fact_q` в `publish.py`.

**Клиффхэнгер в сериях (2026-07-04):** Part 1/2 заканчиваются не генерик «Follow for Part 2», а конкретным тизером содержания следующей части («Part 2: почему все 30 исчезли за одну ночь») + follow-CTA — curiosity gap про реальную деталь тянет на следующую часть сильнее. Только промпт (`SERIES_CTA` в `generate_series.py`). Откат: вернуть старые формулировки `SERIES_CTA`.

### Loop (петля пересмотра) — вариант A: Claude вет­тит, код собирает
**Это итог долгой итерации — НЕ менять без понимания.** Механизм:
1. Claude НЕ пишет loop-строку. Вместо этого отдаёт поле `loop_connectors` — список из {why/how/when/where/because}, для которых «<слово> <хук>» читается как **связное предложение** (вет­тит, зная свой хук).
2. Код (`_append_loop`) **детерминированно** дописывает фразу-петлю в конец скрипта: берёт коннектор из списка Claude, случайную формулировку под него из `CFG["loop_phrases"]` (на языке канала), приклеивает.
3. На стыке петли «<фраза> <хук>» = одно связное предложение. Пример: хук «This fluid in your body changes its entire chemical makeup…» + конец «And this is how.» → петля «And this is how this fluid in your body changes…» ✓.

**Почему так:** форма гарантирована (эмитим только из списка Claude), смысл вет­тит модель на этапе генерации, мусор пройти не может. Прошлые подходы (кастомный фрагмент / общее слово / повтор слов хука) разваливались на стыке по смыслу.

**Петля НЕ на каждом видео:** `loop_probability` (0.5) — настоящая вероятность Бернулли (`random.random()`), а не «каждый второй»; ~50% видео идут с обычной концовкой, чтобы приём не стал формулой. Каждое видео тегается `loop-yes`/`loop-no` → сравнение в `analytics_retention.py`. Решаем по данным, а не по теории.

**Петля ТОЛЬКО в daily Shorts.** Loop-инструкция вынесена в `LOOP_INSTRUCTION` и добавляется к системному промпту только в `generate_script()`. Series и longform используют `BASE_SYSTEM_PROMPT` без неё — у них своя концовка/CTA, петли нет. У серий также отключён визуальный loop-хвост (см. раздел про серии).

### Batch API preload (`prepare_batch.py` + `script_queue.py`) — экономия на генерации сценария
**Сознательно суженный объём:** батчится ТОЛЬКО `generate_script` (Sonnet, самый дорогой одиночный вызов) — не vision-отбор клипов, не TTS, не сборка/загрузка. Anthropic Message Batches даёт **−50%** на этом вызове, но результат готов до 24ч, а не мгновенно — несовместимо с публикацией «сейчас». Поэтому: ночной воркфлоу `prepare-batch.yml` заранее готовит `queue_<channel>.json` (FIFO, держим ~2 дня вперёд, `QUEUE_TARGET=10`), `pipeline.py` при запуске сначала пробует взять сценарий из очереди — **если она пуста, генерит вживую, как раньше** (`generate_script()`). Отсутствие/сбой prepare-джобы **не ломает публикацию** — просто нет экономии в этот раз.

`daily.yml`/`daily-es.yml` коммитят обновлённую очередь обратно в репо после каждого запуска (`permissions: contents: write`) — без этого локальный `pop()` терялся бы вместе с раннером, и следующий прогон снова видел бы несдвинутую очередь из git.

### Ротация опенеров заголовка (2026-07-03, `TITLE_OPENERS` в `generate_script.py`)
Проверено эмпирически (не спекуляция): **84% заголовков EN-канала** начинались с «The»/«Your»/«This» (64 заголовка: 61%+22%+2%). Причина — заголовок генерится в том же вызове, что `hook_template`, и, судя по всему, эхует структуру спич-хука ("This X can...", "You've..."). Модель теперь дополнительно репортит `title_opener` — один из `TITLE_OPENERS` (`the-x`/`your-x`/`scientists-discovered`/`shouldnt-exist`/`nobody-expected`/`only-place`/`question`/`other`), тегируется `opener-<id>`, как `hook-<template>`.

**Дата-driven anti-repeat (`_title_variety_note`):** если ≥60% последних 10 опубликованных заголовков начинаются с "the"/"your"/"this" (эвристика по первому слову, работает и для старых видео без тега) — в промпт добавляется явное требование использовать другой опенер для ЭТОГО видео. Не квота на всё время, а реакция на текущий перекос — если перекос уйдёт, подсказка перестанет добавляться сама. Работает в `generate_script()` и `prepare_batch.py` (один расчёт на весь батч, `past_titles` не меняется внутри одного прогона). В `recycle_winners.py` тегирование есть, но variety-note не подключена (там нет доступа к `past_titles` дешёво — не критично при 3 видео/неделю).

Сравнение retention по опенеру — секция «Опенер заголовка» в `weekly_report.py`. Откат: убрать `TITLE_OPENER_INSTRUCTION` из `_build_user_content` и нормализацию `title_opener` — тег просто перестанет проставляться.

### Outlier-анализ по нише (2026-07-03, `discover_niche_topics.py` + `discover-niche.yml`)
Индустриальные данные: каналы, ищущие видео-выбросы у **чужих** каналов в нише (не только свою статистику), растут в 2-3× быстрее — сигнал темы, не факта, без копирования контента. Раз в неделю (Пн 09:37 UTC, GitHub-cron) по каждой теме из `TOPICS_POOL` (14 тем) идёт `search().list()` (10 самых просматриваемых Shorts), затем дешёвый батч `videos.list`/`channels.list` за просмотрами/подписчиками.

**Запрос строится на языке канала** (`_topic_query_term()`): EN — тема как есть (уже на английском в `TOPICS_POOL`), например `"ancient history facts shorts"`. ES — испанский перевод из `CFG["playlist_titles"]` (естественная фраза, не ALL-CAPS слово, как в `topic_cta_words`) с убранным префиксом «Datos de/del/en », например `"Historia Antigua datos curiosos shorts"`, а не смешанный `"ancient history datos curiosos shorts"`. Баг найден при первом реальном прогоне (2026-07-03): смешанный язык давал мало/нерелевантных результатов по части тем (0-5 вместо 10). Нет перевода темы (`future technology` изначально отсутствовал в `topic_cta_words`, но есть в `playlist_titles`) — фолбэк на английскую тему.

**Метрика "выброс":** `views / subscribers >= 15.0` (и `views >= 20 000`, чтобы не ловить шум крошечных каналов) — видео разошлось СИЛЬНО шире собственной аудитории канала. Отличается от «у канала на 20 млн подписчиков и так много просмотров» (это норма для его размера, ratio ~1-2, не выброс) — маленький/средний канал с внезапным хитом (ratio 25-40×) это реально сигнал темы.

⚠️ **Пороги подняты 2026-07-03 (были 5.0/5000)** — первый прод-прогон показал, что мягкий порог 5.0 слишком легко проходит на Shorts (там ratio 4-8× — обычное дело даже без реального "выстрела" темы, алгоритм и так раздаёт Shorts widely): **все 14 тем на EN упёрлись в потолок бонуса** (`count>=3` у каждой), механизм не различал темы между собой. С порогом 15.0/20000 на теме "the ocean" из тех же 10 видео выбросами остаются только 3 (25.7×, 25.5×, 39×) вместо 4 — отсекает пограничный случай 8.0×.

**Как влияет на генерацию:** результат — `{topic: outlier_count}` в `niche_signal_<channel>.json` (коммитится воркфлоу). `_pick_topic()` (`generate_script.py`) добавляет мягкий множитель веса `×(1 + 0.15 × min(count, 3))` — максимум ×1.45 при 3+ выбросах, чтобы один аномальный чужой ролик не перевешивал всю квоту 70/20/10, только подталкивал тему выше в ранжировании. Нет файла/данных — множитель 1.0, поведение как раньше.

Квота: `search().list` = 100 ед/запрос × 14 тем = 1400 ед/канал за прогон (раз в неделю) — безопасно даже при текущем ежедневном расходе на публикацию. Откат: задизейблить `discover-niche.yml` в Actions — `_load_niche_signal()` вернёт пустой словарь, вес не изменится.

### Пороги retention (2026-07-02, `analytics_retention.retention_threshold`)
YouTube Shorts работает в режиме "explore-and-exploit": каждый новый Short тестируется на маленькой аудитории, и если retention падает ниже порога — раздача обрывается почти сразу, а не постепенно снижается. Индустриальные 2026-бенчмарки (не официальный YouTube-документ, но воспроизводимая цифра в нескольких независимых источниках): **65% для <30с, 50% для 30-60с**. `weekly_report.py` теперь считает не абстрактное "выше/ниже среднего", а конкретный pass/fail по этому порогу — секция «Порог retention: N/M видео прошли» + список худших с недостающим %.

### Валидация-гейт (`_validate` + `_better` в `generate_script.py`)
После генерации — **один** таргетный повтор, только если плохо (платный 2-й вызов; по замеру на 30 запусках EN daily.yml 2026-06-26…07-01 — **8/30 (~27%)**, все 8 из-за длины, ни одного по loop_connectors):
- слов > 93 (`SCRIPT_MAX_WORDS`) → длинно (повтор регенерирует и `video_queries`, десинхрона нет)
- `loop_connectors` пустой/невалидный

Выбор лучшего из двух (`_better`): меньше проблем; **при равенстве — короче скрипт** (иначе при «обе длинные» оставался бы длинный оригинал — это давало 48с).

**Самопроверка длины (2026-07-01):** модель обязана в поле `word_count` посчитать слова готового скрипта перед ответом. **Важно:** промпт явно просит считать **молча** — первая версия просила "draft the script, COUNT it, trim, recount" прямо в тексте ответа, из-за чего модель писала видимые рассуждения перед JSON и упиралась в `max_tokens` (не успевала дойти до `{`, 2/10 EN и 9/10 ES заявок в первом батч-прогоне не распарсились). Исправлено на "silently count... do NOT show any draft/reasoning", `max_tokens` поднят 1200→1600 про запас. `_validate` логирует, если самоотчёт разошёлся с реальным подсчётом (`abs(reported - actual) > 5`) — для мониторинга, не блокирует. Гейт `SCRIPT_MAX_WORDS` 90→93 (пограничные 91-93 слова не хуже прежних 88-90, но не ретраятся). Откат: убрать `word_count` из промпта/схемы и вернуть `SCRIPT_MAX_WORDS=90` — retry-логика и остальной пайплайн не затронуты.

**Дедуп после батч-генерации (2026-07-01, `prepare_batch.py`):** запросы Batch API идут параллельно и не видят друг друга — модель может независимо выбрать один и тот же канонический факт дважды под разными заголовками (пойманы вживую: армия Камбиса II, озеро Натрон — заголовки разные, факт один). `_is_duplicate()` сравнивает **собственные имена и числа из `script`** (не слова заголовка — те слишком шаблонные вида "El X que Y" и ложно совпадают у разных фактов), ≥2 общих — дубль; резервно — почти дословное совпадение заголовка. Дубликат просто не добавляется в очередь (лог + слот останется недосозданным до следующего ночного прогона).

**28–35 сек (70–88 слов, до 93 без retry)** — топовые факт-Shorts кластеризуются на 25-35с (по ресёрчу алго-2026: retention/AVD — главный сигнал, >45с резкий спад). Гейт `SCRIPT_MAX_WORDS=93`, голосового CTA нет. edge-tts +5% ≈ 2.6 слова/сек. `TTS_MIN_DURATION=20с` (короткий валидный скрипт не ловит ложный retry). ### Что в контенте работает НЕ идеально (на будущее)

### Retention-кривая по секундам (2026-07-02, `analytics_retention._retention_curve` + `biggest_drop`)
`averageViewPercentage` — одно число на видео, не показывает, **в какой момент** зритель отваливается (на хуке? на reveal? на twist?). YouTube Analytics отдаёт посекундную кривую через `elapsedVideoTimeRatio`/`audienceWatchRatio` — но эндпоинт принимает фильтр только по **одному** `video==` за раз (не батчится, как обычный `_retention`), поэтому тянем её точечно, не для всех 50 видео разом.

**В weekly_report.py:** для `DROP_OFF_SAMPLE=2` худших по retention видео недели (только с реальными данными, `pct>0`) тянется кривая и находится сегмент с наибольшим падением `audienceWatchRatio` → `biggest_drop()` возвращает примерную секунду обрыва и величину падения в п.п. Секция «📉 Где теряем зрителя» в отчёте — не абстрактное «эта тема слабая», а «на 6-й секунде теряем 25 п.п. аудитории» — конкретное место в структуре сценария для разбора. Сбой на одном видео не рушит остальной отчёт (`try/except` внутри `_add_drop_offs`).

⚠️ **Проверено на проде (2026-07-02) — на текущем масштабе данных нет.** У `elapsedVideoTimeRatio` есть недокументированный официально, но воспроизводимый минимальный порог объёма сессий просмотра (защита анонимности зрителей) — на видео с 2500+ просмотров и 5+ днями жизни API вернул валидную схему, но 0 строк. `averageViewPercentage` (агрегат) при этом работает нормально при любом объёме. Код безопасен — при пустой кривой секция просто не появляется в отчёте, ничего не падает и не врёт (`biggest_drop([], ...)` → `None`). Когда просмотры/канал вырастут — функция сама начнёт отдавать данные, код трогать не нужно. До тех пор считать эту секцию отчёта неактивной, а не «фича не работает».

**Ручной разбор конкретного видео:** `python analytics_retention.py <video_id>` — печатает всю кривую по 1%-долям длины + самый большой обрыв.

### План проверок по данным (`analytics_retention.py`)
Решения по теории отложены до данных — НЕ менять вслепую:
- **`loop_probability`** (сейчас 0.5): данные пока неубедительны (5 видео с петлёй против 2 без, ~74% vs ~67% retention) — выборка мала, ждать больше данных.
- **`hook-<template>` веса** — тегирование только что включено (daily+series); когда накопится статистика, взвесить сильнейшие шаблоны так же, как темы.
- **`future technology`** — возвращена в EN-пул на ре-тест (старые провальные видео удалены → тема с нулевыми данными получает нейтральный вес, честный шанс под текущим промптом). Если снова <300 просмотров — убрать окончательно.
- **Длина видео** (сейчас 28-35с) и **хук-плашка vs интрига** — та же логика, ждём данных.

## История ключевых решений

- **Ниша "факты"** — выбрана из-за низкого риска авторских прав и простоты генерации
- **Английский язык** — больше аудитория, выше CPM рекламодателей
- **Pollinations.ai → стоковые клипы** — первая версия использовала AI-генерацию картинок с эффектом Ken Burns, потом переключились на реальные видео Pexels/Pixabay (лучше смотрится)
- **MoviePy 1.x закреплён** в requirements — 2.x сломал совместимость API (`moviepy.editor` убрали)
- **Python 3.12** — установлен на машине взамен неполной установки Python 3.13
- **ImageMagick** нужен для субтитров; на Windows ищет `magick.exe` (не `convert.exe`) — путь прописан явно в `build_video.py`
- **Reddit-интеграция** — отменена: Reddit в 2025-2026 требует одобрения приложения и запрещает авто-постинг ссылок ("Responsible Builder Policy")
- **Расписание** подстроено под пики активности US Eastern: публикуем за 2-3 часа до пиков (12:00-15:00 и 18:00-21:00 ET). При переходе США на зимнее время (ноябрь) — сдвинуть расписание на час раньше
- **"inauthentic content"** — YouTube с 2025 усилил детекцию шаблонного контента. Поэтому сделана ротация: голоса TTS, зум фона, цвет субтитров, текст CTA
- **Названия каналов (60SecFacts / Datos en 30s)** — намеренно НЕ меняются, хотя реальная длина роликов (28-35с) не совпадает с «60/30» в имени: число работает как бренд-якорь, не буквальный таймер («60 Minutes» тоже не идёт 60 минут); риск смены (ссылки/URL/бренд-узнаваемость) не оправдан отсутствием доказанного вреда

## cron-job.org — внешний триггер

GitHub Actions cron ненадёжен (опаздывает на 10-40 мин, иногда пропускает). Поэтому daily.yml/daily-es.yml триггерятся **только** через [cron-job.org](https://cron-job.org) — это теперь ОСНОВНОЙ (единственный) триггер для них, а не дублёр. Родной GitHub-`schedule` у них убран (плодил дубли — см. раздел «Расписание»).

**Как работает:**
1. cron-job.org по расписанию отправляет POST-запрос на GitHub API
2. GitHub получает `workflow_dispatch` событие и запускает workflow
3. Слоты бьют точно вовремя (cron-job.org не опаздывает как GitHub-cron)

**Задания** (Moscow time = UTC+3). Daily-задания — каждый день; серии — только Пн/Вт/Ср; лонгформ — только Вс:

| Название | Workflow | Дни | UTC | Moscow |
|----------|----------|-----|-----|--------|
| EN-1300 | daily.yml | ежедн | 13:07 | 16:07 |
| EN-1600 | daily.yml | ежедн | 16:13 | 19:13 |
| EN-2000 | daily.yml | ежедн | 20:07 | 23:07 |
| EN-2300 | daily.yml | ежедн | 23:07 | 02:07 |
| EN-0000 | daily.yml | ежедн | 00:07 | 03:07 |
| ES-1300 | daily-es.yml | ежедн | 13:17 | 16:17 |
| ES-2000 | daily-es.yml | ежедн | 20:17 | 23:17 |
| ES-0000 | daily-es.yml | ежедн | 00:17 | 03:17 |
| EN-SERIES | weekly-series.yml | Пн+Вт+Ср | 00:07 | 03:07 |
| ES-SERIES | weekly-series-es.yml | Пн+Вт+Ср | 00:17 | 03:17 |
| EN-LONG | weekly-longform.yml | Вс | 23:00 | 02:00 |
| ES-LONG | weekly-longform-es.yml | Вс | 23:17 | 02:17 |

**Важно про серии в 00:07/00:17:** в эти дни daily-задание (EN-0000/ES-0000) тоже сработает, но guard в `daily*.yml` пропустит обычный шортс (`HOUR=00`, Пн/Вт/Ср) — выйдет только серия.

**Если нужно обновить токен в cron-job.org:**
Зайди на cron-job.org → каждое задание → Edit → в заголовке `Authorization` замени токен.

⚠️ **Таймзона каждого задания должна давать правильное время в UTC** (весь код считает в UTC). Daily-задания стоят в `Europe/Moscow` со **скомпенсированным** временем (16:07 МСК = 13:07 UTC и т.п.) — это ок. Но был реальный баг: `ES-SERIES`/`ES-LONG` стояли в `Europe/Moscow` **без** компенсации → серия уезжала на 21:17 UTC вместо 00:17, теряя прайм-слот и коллизируя с ES-2000. Пофикшено — обе таймзоны переставлены на **UTC** напрямую. Проверяя/меняя таймзону в форме: после выбора **обязательно жать SAVE**, не уходя со страницы — иначе выбор откатывается при следующей загрузке.

## Telegram-бот «расписание»

Отдельно от `notify.py` (алерты о выходе видео) — бот отвечает на слово **«расписание»** слотами выхода (Вьетнам/Москва) и обратным отсчётом до следующего ролика.

**Прод: Cloudflare Worker** (`telegram-bot/schedule_worker.js`) — работает 24/7, ПК не нужен. Деплой через дашборд Cloudflare (см. `telegram-bot/README.md`): создать Worker → вставить код → добавить секрет `BOT_TOKEN` → привязать `setWebhook`. Слоты захардкожены в JS (`EN_SLOTS`/`ES_SLOTS`) — при смене расписания поправить и передеплоить.

**Запасной вариант:** `src/schedule_bot.py` — локальный long-polling, работает только пока запущен на ПК. Webhook и polling **несовместимы одновременно** (после `setWebhook` локальный поллер перестаёт получать апдейты) — используется только если Worker недоступен.
